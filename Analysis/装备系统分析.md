# 装备系统核心逻辑分析（脚本层）

## 结论先行

- 装备系统脚本层的核心不是“直接换手上模型”，而是 **`QueueRequest -> 解析动作 -> 发给PSM装备状态机`**。
- `EquipmentSystemPlayerData.OnEquipmentSystemWeaponManipulationRequest()` 是武器语义请求的核心汇聚点。
- “切槽位后武器自动拿在手上”是系统设计结果：槽位请求最终会下发 `Equip` 到左右手装备状态机。

---

## 1) 系统架构：脚本层真正可控的对象

### 1.1 核心系统与入口

- 主系统：`EquipmentSystem`
- 常见入口：
  - `GameInstance.GetScriptableSystemsContainer(game).Get(n"EquipmentSystem") as EquipmentSystem`
- 执行方式：`QueueRequest(request)`（异步排队）

### 1.2 请求模型分层

- **语义层请求（推荐用于战斗切换）**
  - `EquipmentSystemWeaponManipulationRequest`
  - 通过 `requestType` 表达“我要做什么”（切下一把、切槽1、收枪、切拳头等）
- **槽位层请求（偏管理/指定）**
  - `EquipmentSystemPlayerEquipRequest`、`UnequipRequest`、`SetActiveItemInEquipmentArea`
  - 通过 `itemID/slotIndex` 直接指定“我要哪件物品在哪个槽”

> 结论：实战里优先走语义层，除非你确实需要硬指定 `ItemID`。

---

## 2) 核心函数：`OnEquipmentSystemWeaponManipulationRequest`

这是脚本层武器操作的主调度函数，流程可概括为：

1. **判定请求类型**
   - 是否是卸下类请求（`isUnequip`）
2. **动作转物品**
   - `GetItemIDfromEquipmentManipulationAction(requestType)`
3. **门控检查**
   - 热键限制、item有效性、库存存在性、玩法限制
4. **特殊分流**
   - 激活型义体可能直接 `UseItem`（而不是装备流程）
5. **标准装备/卸载路径**
   - 计算左右手槽位 `requestSlot`
   - 更新 `slotActiveItem` / `lastUsed` / `activeIndex`
   - `SendPSMWeaponManipulationRequest(Equip|Unequip, requestSlot, animType)`

### 2.1 为什么它是“核心”

因为它同时完成三件事：

- 语义解释（Action -> Item）
- 状态维护（active slot / last used）
- 执行下发（PSM装备状态机）

### 2.2 切枪速度真正落地位置（`EquipmentBaseTransition`）

虽然装备请求入口在 `EquipmentSystemPlayerData`，但“切得快不快”主要在 PSM 装备状态机里落地：

- 文件：`DecompiledGameScripts/cyberpunk/player/psm/equipment.swift`
- 关键函数：
   - `EquipmentBaseTransition.GetWeaponEquipDuration(...)`
   - `EquipmentBaseTransition.GetWeaponUnEquipDuration(...)`

这两个函数最终读取武器实体 Stat：

- `gamedataStatType.EquipDuration`
- `gamedataStatType.UnequipDuration`

并把时长写入装备动画特征（如 `AnimFeature_EquipUnequipItem` / `AnimFeature_EquipType`），再驱动实际装备/卸载节奏。

因此，`OnEquipmentSystemWeaponManipulationRequest` 决定“切什么”，而 `EquipmentBaseTransition` 决定“切多快”。

> 补充：`equipAnimType` 为 `Instant` 或部分特殊条件（如投掷冷却分支）可把时长压到 `0.00`，属于状态机层面的快速通道。

---

## 3) Action 到 Item 的核心映射机制

`GetItemIDfromEquipmentManipulationAction()` 负责把动作语义映射为最终目标物品，例如：

- `CycleNextWeaponWheelItem` -> `CycleWeapon(true, false)`
- `RequestWeaponSlot1~4` -> `GetWeaponSlotItem(1~4)`
- `RequestNextThrowableWeapon` -> `GetNextThrowableWeapon()`

这层映射是 MOD 最稳妥的切入点之一（比直接改底层状态机风险低）。

---

## 4) 为什么“切槽位后会自动拿在手上”

以 1~4 数字键为例：

1. 上半身状态机监听 `WeaponSlot1~4` 输入（`upperBodyTransitions.swift`）
2. 触发 `SendEquipmentSystemWeaponManipulationRequest(...RequestWeaponSlotX...)`
3. 请求进入 `OnEquipmentSystemWeaponManipulationRequest`
4. 解析到具体 `item`（`GetWeaponSlotItem`）
5. 执行 `SendPSMWeaponManipulationRequest(Equip, requestSlot, ...)`

**关键点在第5步**：系统明确向手部装备状态机发送了 `Equip`，所以它不是“仅切UI高亮槽位”，而是“切槽并装备到手”。

---

## 5) 执行链路（从输入到落地）

### 5.1 通用链路

`输入/事件 -> WeaponManipulationRequest -> EquipmentSystem.QueueRequest -> OnEquipmentSystemWeaponManipulationRequest -> SendPSMWeaponManipulationRequest -> 右手/左手装备状态机`

### 5.2 与 UI/状态机关系

- `UpdateEquipAreaActiveIndex`、`SetLastUsedItem`：维护轮盘和“上次武器”逻辑
- PSM事件：决定动画、时机、是否可执行
- 因此行为会受当前状态（上半身状态、限制状态、战斗/载具状态）影响

## 6) 场景补充（作为验证样例）

### 6.1 投掷回收中的“下一把可投掷”

`MeleeThrowReloadEvents.EquipNextWeapon()` 发送 `RequestNextThrowableWeapon`：

- 先选下一把可投掷武器（`GetNextThrowableWeapon`）
- 找不到则回落到普通循环武器
- 最终仍走标准 Equip 下发

这说明该场景只是核心机制的一种特例，而不是独立流程。

### 6.2 1~4 槽位切换

`RequestWeaponSlot1~4` -> `GetWeaponSlotItem(1~4)` -> `Equip` 下发。

槽4通常会落到近战义体/拳头（`GetMeleewareOrFistsItemID()`）。

### 6.3 按 B 键收纳/拔出武器（`HolsterWeapon`）

在上半身状态机 `upperBodyTransitions.swift` 里，B键对应动作名是 `HolsterWeapon`，并且“收纳”和“拔出”分别发生在不同上半身状态：

- **当前手上有武器（SingleWield/ForceSafe链路）**
   - `UpdateSwitchItem()` 中检测：`IsActionJustTapped(n"HolsterWeapon")`
   - 若没有 `FirearmsNoUnequip` 限制：
      - 发送 `EquipmentManipulationAction.UnequipWeapon`
      - 进入标准卸下链路（最终 `SendPSMWeaponManipulationRequest(Unequip, ...)`）

- **当前空手（EmptyHands链路）**
   - `EmptyHandsEvents.OnUpdate()` 中把 `HolsterWeapon` 纳入“请求拔武器”的触发条件
   - 会选择一个“可拔出的目标请求”（如 `RequestLastUsedOrFirstAvailableWeapon` / 受状态标签影响的变体）
   - 若是“按住触发显示武器”（`IsActionJustHeld(n"HolsterWeapon")`），会带 `gameEquipAnimationType.FirstEquip`

所以 B 键不是单向“仅收枪”，而是：

- 有武器时倾向于发 `UnequipWeapon`（收纳）
- 空手时倾向于发 `RequestLastUsedOrFirstAvailable...`（拔出）

本质仍是同一条主链路：`输入 -> ManipulationAction -> OnEquipmentSystemWeaponManipulationRequest -> PSM Equip/Unequip`。

## 7) 能力边界与MOD建议

### 7.1 能力边界

- `QueueRequest` 是排队执行，不是同步立即生效。
- 请求是否成功受状态机与玩法限制共同约束。
- 部分义体请求会分流到 `UseItem`，不一定走装备动画链。

### 7.2 实战建议

1. 优先改 Action->Item 映射，少碰深层状态机。
2. 若目标是“加快切枪速度”，优先改 `EquipDuration/UnequipDuration`（数据层或按武器类型做倍率）。
3. 需要“仅切槽不出手”时，重点关注并拦截 `SendPSMWeaponManipulationRequest(Equip, ...)` 的触发条件。
4. 始终保证 `request.owner` 正确，否则请求可能静默失效。

---

## 一句话总结

装备系统脚本层的核心是：

**用语义请求驱动装备决策，再由 `OnEquipmentSystemWeaponManipulationRequest` 将结果下发到 PSM 装备状态机执行。**

所以槽位切换后自动出手，是这条主链路的“预期行为”。
