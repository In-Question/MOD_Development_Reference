# 近战武器投掷与回收机制分析（基于反编译脚本）

## 结论先行

- 游戏里“回收”本质上是 **ThrowRecovery 资源池冷却恢复 + 武器重新可见/可用**，并不等同于“飞出去再自动飞回手里”的回旋镖逻辑。
- 投掷武器流程主要由玩家近战状态机驱动（`meleeTransitions.swift`），实体飞行与拾取由投掷物脚本驱动（`meleeProjectile.swift`）。
- 回收完成的核心判定是：`ThrowRecovery` 到达最大值（或被强制置满），随后退出 `MeleeThrowReload` 状态并恢复武器可见。

---

## 1) 关键脚本与职责

### 状态机与动作触发

- `DecompiledGameScripts/cyberpunk/player/psm/meleeTransitions.swift`
  - `MeleeThrowAttackDecisions`：判断能否进入投掷攻击。
  - `MeleeThrowAttackEvents`：真正发射投掷物、设置投掷后冷却状态。
  - `MeleeThrowReloadDecisions / Events`：处理投掷后的“回收/装填”阶段。
  - `MeleeTransition.IsThrownWeaponReloading()`：以 `ThrowRecovery` 是否满值判定是否仍在回收中。

### 投掷物行为

- `DecompiledGameScripts/cyberpunk/projectiles/meleeProjectile.swift`
  - `MeleeProjectile`：飞行、碰撞、落水、交互拾取。
  - 初始化时将武器 `ThrowRecovery` 置最小值（开始冷却）。
  - 交互拾取时将 `ThrowRecovery` 置最大值（立即回收完成）。

### UI反馈

- `DecompiledGameScripts/cyberpunk/UI/weapons/crosshairs/melee/crosshair_melee_knife.swift`
  - 监听 `ThrowRecovery`，刷新刀具回收条和淡入淡出动画。

### 装备与换刀

- `DecompiledGameScripts/cyberpunk/player/psm/equipment.swift`
  - 若 throwable 武器冷却未满，装备动画会带 `isThrowReloading`。
- `DecompiledGameScripts/cyberpunk/systems/equipmentSystem.swift`
  - `GetNextThrowableWeapon()`：回收期按攻击输入切下一把可投掷近战。

### 玩家状态效果（天赋/词条导致的“秒回收”）

- `DecompiledGameScripts/cyberpunk/player/player.swift`
  - 处理多个 tag（如 `JugglerPerkRemoveKnifeCooldown`、`ThrowMod1RemoveCooldown`）时，直接把 `ThrowRecovery` 设为满值。

## 2) 投掷触发链路

### 2.1 进入瞄准与投掷判定

1. 玩家按住格挡进入 `MeleeTargeting`（投掷瞄准态，要求可投掷条件满足）。
2. 在瞄准态按下攻击，`MeleeThrowAttackDecisions.EnterCondition()` 继续检查：
   - 有投掷攻击记录；
   - 当前仍是格挡按住；
   - `MeleeAttackPressed`；
   - 若武器有 `Throwable` 标签，还需 `CanThrowWeapon` 为真。

### 2.2 发射时机

`MeleeThrowAttackEvents.OnUpdate()` 在达到 `attackEffectDelay` 且未发射过时：

- `SetIsThrowReloading(true)`
- `SpawnMeleeWeaponProjectile()` 生成投掷物
- `ApplyThrowableCooldown()` 读取武器 `gamedataStatType.ThrowRecovery` 写入动画参数
- `weapon.SetVisible(false)` 隐藏手中武器模型

这就是“刀扔出手后本体隐藏”的直接来源。

---

## 3) 回收机制本体（ThrowRecovery 资源池）

### 3.1 冷却开始

`MeleeProjectile.OnProjectileInitialize()`：

- 对应武器实体调用
  `RequestSettingStatPoolMinValue(..., ThrowRecovery, ...)`
- 相当于把回收进度条清空，标记“正在回收”。

### 3.2 冷却结束的两种路径

1. **自然恢复到满值**（由资源池系统按配置恢复）
   - `ThrowingMeleeReloadListener.OnStatPoolMaxValueReached()` -> `TryToReleaseProjectile()`
2. **玩家拾取/效果强制置满**
   - `MeleeProjectile.OnInteractionActivationEvent()`：交互拾取时直接请求 `ThrowRecovery` 满值。
   - 玩家状态效果（杂耍者、特定词条等）也会在 `player.swift` 中直接置满。

### 3.3 状态机退出回收态

`MeleeThrowReloadDecisions.ExitCondition()`：

- 若 `IsThrownWeaponReloading()` 仍为真（即池未满）=> 不退出
- 或在最短回收时间 `Items.MeleeWeapon.minimumReloadTime` 内 => 不退出
- 两者都满足“可退出”后，离开 `MeleeThrowReload`

`MeleeThrowReloadEvents.OnExit()`：

- `SetIsThrowReloading(false)`
- 移除相关音频触发
- `weapon.SetVisible(true)` 恢复武器可见

---

## 4) 投掷物生命周期细节

`meleeProjectile.swift` 中可见：

- 支持水面碰撞、落水下沉、弹跳/穿透/粘附判定（`ThrowingMeleeCollisionEvaluator`）。
- 命中后可激活交互事件供玩家拾取。
- `TryToReleaseProjectile()` 只执行一次（`m_wasPicked` 防重入），用于清理投掷物。

这说明“回收”在实现上是**投掷物生命周期结束 + 武器冷却恢复**的组合，而不是同一实体往返。

---

## 5) UI与交互表现

- 准星状态：`Targeting` 时进入投掷准星动画，`ThrowAttack`/回收时切 `knifeReloading`。
- 回收条：监听 `ThrowRecovery` 值变化。
  - 小幅变化：更新环形进度。
  - 达到满值：播放结束淡出。
  - 到最小值：显示并播放开始淡入。

玩家主观感知就是“扔出 -> 冷却圈转动 -> 可再次投掷/切换结束”。

---

## 6) 对 MOD 的可用切入点

1. **改回收时长**
   - 改 `gamedataStatType.ThrowRecovery` 数值来源（武器/词条/状态效果）。

2. **改回收完成条件**
   - 在类似 `player.swift` 的状态效果流程里，按条件调用 `RequestSettingStatPoolMaxValue` 实现“击杀返还/暴击返还”。

3. **改投后行为**
   - `MeleeThrowReloadEvents.OnUpdate()` 可扩展为更激进的自动切刀逻辑。

4. **做“伪自动回旋”**
   - 现有脚本没有标准回旋返回轨迹；可通过“更快置满 `ThrowRecovery` + 特效/音效 + 提前恢复可见”实现体感上的自动回收。

---

## 7) 机制总结（一句话）

近战武器投掷回收机制的核心不是“物理回飞”，而是：

**状态机触发投掷 -> 投掷物生成并把 `ThrowRecovery` 置空 -> 资源池恢复/拾取/天赋置满 -> 退出回收态并恢复武器可见与可用。**

## 8) 补充：未回收时按左键为何会触发“下一把武器拔刀攻击”

在反编译脚本中可串出完整链路：

1. `MeleeThrowReloadEvents.OnUpdate()` 在投掷回收态监听攻击输入（左键）。
2. 满足 `Items.MeleeWeapon.weaponSwapOnAttackDelay` 后，会调用 `EquipNextWeapon()`。
3. `EquipNextWeapon()` 发送 `EquipmentManipulationAction.RequestNextThrowableWeapon`。
4. `EquipmentSystem.GetNextThrowableWeapon()` 会优先找下一把带 `Throwable` 标签的武器；
  若找不到，则 **回退到** `CycleWeapon(true, false)`，即普通“下一把武器”逻辑。
5. 若被切到的是武士刀等近战武器，进入装备流程后，`MeleeEquippingDecisions` 会检查该武器是否有 `MeleeEquipAttack` 记录；
  输入满足时可转入 `ToMeleeEquipAttack()`，最终由 `MeleeEquipAttackEvents` 执行 `EMeleeAttackType.Equip`。

因此你看到的现象本质是：

**“回收态左键请求下一把可投掷” -> “找不到则回退普通切刀” -> “新刀在装备态触发 EquipAttack”。**

## 9) 补充：为什么用 R 键“卡回收”后，左键反而难以触发下一把刀的 EquipAttack

你当前 CET 改法（`Override("MeleeThrowReloadDecisions", "ExitCondition", ...)`）的关键行为是：

- 先执行原始 `ExitCondition`（`baseReady = wrappedMethod(...)`）
- 只有 `baseReady == true` 且 `Reload` 按下才允许退出

这意味着：

1. 你并没有把“真实未回收”一直保持住。  
  因为 `baseReady == true` 的前提本来就是原生退出条件成立（包括 `ThrowRecovery` 已到退出门槛）。

2. 你真正卡住的是“状态机退出”，不是“资源池恢复进程”。  
  即角色被人为停在 `MeleeThrowReload`，而不是保持在原生那种“资源池未满导致的未回收”。

3. `EquipAttack` 的触发窗口在 `MeleeEquipping` 相关决策里，而你把状态机长时间钉在 `MeleeThrowReload`，会破坏这段输入时序。  
  另外 `MeleeThrowReloadEvents` 里 `m_isSwitchingWeapon` 是一次性锁存，首次置 `true` 后本状态内不会重复发起切武器，也会放大“按了没反应”的体感。

所以你的判断方向应调整为：

- 不是“只卡了回收没卡状态机”；
- 更接近 **“卡住了状态机退出（且时机在原生可退出之后），导致后续 EquipAttack 所需的状态/输入窗口被错位”**。

一句话：

**原生可用的是“真实未回收态下的换刀时序”；你现在实现的是“已满足原生退出后的人为驻留态”，两者在 PSM 输入窗口上不是同一个状态。**
